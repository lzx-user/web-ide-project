### 🗺️ Web IDE 前端架构与全链路流转笔记

#### 1. 🏗️ 组件之间的层级与联系（树状结构）

在 React 中，组件是有严格的“父子层级”关系的。我们当前的架构是一个典型的**“星型结构”**（或称为“大管家模式”）。

- **👑 `App.jsx` (父组件 / 智能组件)**
  - **角色**：唯一的“老板”和“数据中心”。
  - **职责**：它不负责具体的页面长什么样，只负责把控所有的全局数据（`状态 State`），以及统筹规划（定义操作数据的方法）。
  - **联系**：它将四大子组件包裹在内部，并通过 **Props（参数传递）** 的方式，把数据和指令下发给各个子组件。
- **👨‍💻 四大子组件 (子组件 / 笨组件)**
  - **角色**：打工人。
  - **职责**：它们只负责“拿钱办事”（接收 Props，渲染 UI），以及“有事汇报”（用户点击了什么，就通过调用 Props 里的函数向 App 汇报）。
  - **相互联系**：**这四个子组件之间是绝对隔离的，它们互不认识，也不能直接通信！** 如果 `Sidebar` 切换了文件，想让 `CodeEditor` 知道，必须通过 `App.jsx` 在中间传话。

#### 2. 🔄 核心状态（State）的分布网

`App.jsx` 手里捏着四张“王牌”（State），分别下发给不同的组件：

1. **`activeFile` (当前文件名)**
   - 👉 发给 `Sidebar`：用于高亮当前文件。
   - 👉 发给 `Header`：用于在顶部显示文件名。
2. **`currentCode` (当前代码内容)**
   - 👉 发给 `CodeEditor`：用于在代码区显示文本。
3. **`isRunning` (是否正在运行)**
   - 👉 发给 `Header`：用于控制“运行”按钮变成灰色并禁用。
4. **`terminalLogs` (终端日志记录)**
   - 👉 发给 `Terminal`：用于在底部一行行画出彩色日志。

#### 3. ⏱️ 完整的执行流程 (以“编写并运行代码”为例)

假设用户现在在网页上敲下了一行代码，并点击了“运行”按钮，整个系统是这样协同运转的：

- **第一阶段：输入同步 (打字流程)**
  1. 用户在界面的中间战区键盘敲击 `console.log(1)`。
  2. `CodeEditor` 监听到按键，立刻调用老板给的汇报电话：`setCode("console.log(1)")`。
  3. `App.jsx` 接到电话，更新自己的状态 `currentCode`。
  4. React 发现状态更新，立刻把最新的 `currentCode` 重新发给 `CodeEditor`，屏幕上就显示出了新代码。（这在毫秒级完成）
- **第二阶段：触发运行 (点击流程)**
  1. 用户点击顶部战区的“▶ 运行”按钮。
  2. `Header` 组件本身不知道怎么运行，它只负责拨打老板的执行专线：触发 `onRunCode()`。
- **第三阶段：App 大管家接管 (核心调度)**
  1. `App.jsx` 收到指令，开始执行 `handleRunCode` 函数。
  2. **状态更新一**：调用 `setIsRunning(true)`。`Header` 收到新状态，按钮立刻变灰并显示“⏳ 运行中...”。
  3. **状态更新二**：调用 `addLog('info', '正在执行...')`，向日志数组塞入一条新数据。`Terminal` 收到新数组，底部立刻出现一行提示语。
- **第四阶段：前后端通信 (网络请求)**
  1. `App.jsx` 将自己保管的 `currentCode` 打包，通过 `fetch` 发送给 Node.js 后端。
  2. 前端进入安静的等待期（此时用户无法重复点击运行按钮，因为 Header 被禁用了）。
- **第五阶段：结果渲染 (收尾流程)**
  1. `App.jsx` 收到了后端返回的执行结果 `{ success: true, output: "1" }`。
  2. **状态更新一**：调用 `addLog('success', '1')`，向日志数组追加真实结果。`Terminal` 组件自动根据 `success` 类型，在底部渲染出一行绿色的 `1`。
  3. **状态更新二**：调用 `setIsRunning(false)`。`Header` 组件收到新状态，运行按钮恢复原状（变绿且可点击）。

#### 💡 核心设计思想总结 (React 哲学)

这种架构被称为 **“单向数据流 (One-way Data Flow)”** 和 **“状态提升 (Lifting State Up)”**。

- **单向数据流**：数据永远是从上（App）往下（子组件）流动的，就像瀑布一样。子组件绝对不能强行修改老板的数据。
- **状态提升**：如果两个平级的组件（比如 Header 和 CodeEditor）需要共享同一个数据（代码内容），就把这个数据“提升”到它们共同的父组件（App）里去集中管理。
