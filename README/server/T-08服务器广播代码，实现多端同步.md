# T-08：服务器广播代码，实现多端同步（解决死循环与防抖）

## 1、开发思路

到了这一步，前端已经能把代码发给后端了，接下来的核心目标是让**所有连上服务器的电脑都能看到代码的实时变化**。这里面藏着协同编辑领域的三个核心痛点，我们的思路也是围绕解决它们展开的：

### **第一步：后端做个“聪明的传线员” (Broadcast)**

当 A 敲击键盘发送了代码，后端不应该把代码广播给“所有人”（如果发给所有人，A 会再次收到自己刚打字的内容，导致光标乱跳）。后端应该只把代码发给**除 A 以外的其他客户端**。

### **第二步：前端装个“防波堤” (Debounce)**

程序员打字的速度是极快的，如果每敲一个字母（哪怕只是一个空格）都触发一次网络请求，服务器会被瞬间打挂，网络延迟也会导致别人看到的画面疯狂抽搐。我们需要让前端“攒一攒”，等用户稍微停顿了再统一发送。

### **第三步：前端安把“拦截锁” (The Echo Loop Lock)**

这是最致命的问题：**死循环**。 B 收到 A 的代码 -> B 的编辑器更新显示 -> B 的编辑器觉得“哎呀我内容变了”从而触发 `onChange` -> B 又把这代码发回给 A -> A 触发 `onChange`... 轰！浏览器直接卡死崩溃。 **思路：** 必须用一把“锁”（标志位）来让编辑器区分：到底是我自己用键盘敲的，还是服务器推过来让我被动更新的。

## 2、实操步骤与代码

### **2.1 后端：精准广播，拒绝回音 (index.js)**

**核心知识点：** * 区分 `io.emit` (全局广播) 和 `socket.broadcast.emit` (排除性广播)。

```jsx
// server/index.js

io.on('connection', (socket) => {
  console.log('A user connected:', socket.id); 

  socket.on('codeChange', (newCode) => {
    // 💡 为什么要这样写？
    // 如果用 io.emit，发送者自己也会收到消息，导致他本地的编辑器二次渲染，光标会莫名其妙闪烁。
    // 使用 broadcast 完美解决：告诉除了我自己之外的所有人“我的代码更新了”。
    socket.broadcast.emit('codeChange', newCode);
  })
})
```

### **2.2 前端：流量削峰，引入防抖 (CodeEditor.jsx)**

**核心知识点：** * `lodash` 的 `debounce` 函数。

- 必须使用 React 的 `useCallback` 钩子包裹防抖函数。

```jsx
// frontend/src/components/CodeEditor.jsx
import { useCallback } from 'react';
import { debounce } from 'lodash';

export default function CodeEditor() {
  // ... 其他代码
  
  // 💡 为什么要这样写？为什么要加 useCallback？
  // 如果不加 useCallback，React 每次重新渲染组件时，都会创建一个"全新"的 debounce 函数，
  // 这会导致上一个定时器失效，防抖完全不起作用（依然会像机关枪一样发请求）。
  // useCallback 保证了组件的整个生命周期里，我们都在使用同一个“秘书”帮我们倒计时 1000ms。
  const emitCodeChange = useCallback(
    debounce((newCode) => {
      socket.emit('codeChange', newCode);
    }, 1000), // 延迟1000毫秒（或400毫秒），等键盘停下再发
    []
  );
  
  // ...
}
```

### **2.3 前端：打破死循环，加入拦截锁 (CodeEditor.jsx)**

**核心知识点：** * 使用 `useRef` 创建同步、不触发重绘的标志位（Flag）。

- 在 `useEffect` 中上锁，在 `onChange` 中检查并解锁。

```jsx
// frontend/src/components/CodeEditor.jsx
import { useRef, useEffect } from 'react';

export default function CodeEditor() {
  const [code, setCode] = useState('// 请在此输入代码...');

  // 💡 为什么要用 useRef 而不是 useState？
  // useState 的更新是异步的，且会触发页面重新渲染。
  // 我们只需要一个“静默”的、改了立刻生效的变量来做判断，useRef 是最佳选择。
  const isRemoteUpdate = useRef(false);

  // 接收端：监听服务器发来的代码
  useEffect(() => {
    const handleReceiveChange = (newCode) => {
      // 🚨 核心逻辑 1：被动更新前，先把门锁上！
      // 明确告诉组件：接下来的代码更新是别人发来的，不是我敲的。
      isRemoteUpdate.current = true; 
      
      // 更新编辑器内容。这步必然会触发下面的 handleEditorChange
      setCode(newCode);
    }
    socket.on('codeChange', handleReceiveChange);
    return () => socket.off('codeChange', handleReceiveChange);
  }, []);

  // 发送端：处理编辑器内容变化
  const handleEditorChange = (value) => {
    // 🚨 核心逻辑 2：检查门锁！
    if (isRemoteUpdate.current) {
      // 如果锁是闭合的，说明这个 onChange 是被服务器推过来的数据触发的。
      // 我们直接拦截，切断死循环！把锁重新打开，准备迎接下一次真实的手动输入。
      isRemoteUpdate.current = false; 
      return; 
    }

    // 如果走到这里，说明锁是开着的（是用户真实敲击键盘）
    setCode(value); // 实时更新本地画面，保证打字丝滑
    emitCodeChange(value); // 调用防抖函数发送给后端
  };
}
```

## 💡 面试必杀技

**👨‍💼 面试官提问：**

> “我看你的项目里实现了多端代码的实时协同。这种协同编辑器很容易遇到‘无限循环（死循环）’和‘性能卡顿’的问题，请问你是如何解决这两个痛点的？”

**👩‍💻 你的满分回答：**

> “这两个问题我都遇到了，并且做了针对性的架构优化。
>
> **关于死循环：** 编辑器协同最可怕的就是‘回音壁效应’。A发给B，B的编辑器更新后会自动触发 `onChange`，又当做新代码发回给A，导致浏览器直接内存溢出卡死。 我的解决办法是**引入拦截锁机制**。在 React 中，我使用 `useRef` 定义了一个 `isRemoteUpdate` 的标志位。当客户端监听到 Socket 推送的代码时，先将标志位设为 `true` 再执行更新；而在编辑器的 `onChange` 事件最顶层，我会优先判断这个标志位，如果是 `true` 则直接 `return` 拦截网络发送，并重置标志位。之所以用 `useRef` 而不用 `useState`，是因为我们需要标志位同步且静默地生效，不能引发额外的重渲染。
>
> **关于性能卡顿：** 如果用户每敲一个字母就发一次 WebSocket，不仅服务器扛不住，页面也会因为高频重绘而掉帧。 我引入了 `lodash` 的防抖（Debounce）函数，将网络请求的动作延迟化。这里的**核心坑点**是，防抖函数必须被 React 的 `useCallback` 包裹并传入空依赖数组 `[]`。如果不包裹，组件每次重绘都会生成新的防抖定时器实例，导致防抖彻底失效。结合后端的 `socket.broadcast.emit` 排除性广播，最终实现了一个丝滑且低延迟的协同体验。”
