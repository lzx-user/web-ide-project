# Web IDE 开发文档 - T-05: 全栈打通与真实代码执行

## 🎯 核心目标

打破前端的“静态页面”限制，搭建 Node.js (Express) 后端服务器。实现前端编辑器收集代码 --> 发送至后端 --> 后端安全执行代码 --> 返回标准输出 (stdout/stderr) --> 前端终端动态渲染的**完整全栈闭环**。

------

## 🏗️ 架构设计思路

### 1. 后端职责 (Server)

- **搭建 API 接口**：使用 Express 搭建 HTTP 服务，并配置 `cors` 中间件解决前后端分离运行带来的跨域问题。
- **文件沙箱与执行**：接收前端传来的字符串片段，利用 Node.js 的 `fs` (文件系统) 模块将其写入一个临时文件 (如 `temp.js`)，然后使用 `child_process` (子进程) 模块调用系统原生的 `node` 命令去执行它。
- **结果捕获**：捕获执行过程中的标准输出 (Console 打印) 和错误信息 (Error)，打包为 JSON 响应给前端。

### 2. 前端职责 (Frontend)

- **状态收集 (Editor)**：监听 Monaco 编辑器的输入事件，将当前最新的代码内容实时保存在 `App.jsx` 的 State 中。
- **网络请求 (Axios/Fetch)**：当点击“运行”按钮时，将当前选中的文件名和代码内容，通过 POST 请求发送给后端 `http://localhost:3000/api/run`。
- **动态终端 (Terminal)**：将后端返回的结果，封装成带有特定 `type` (success/error/info) 的日志对象，追加到 `terminalLogs` 数组中并实时渲染。

------

## 🛠️ 核心代码实现 (后端 Server)

### 1. 后端主程序 (`server/index.js`)

后端的核心逻辑是接收代码、写入文件、执行文件并返回结果。

JavaScript

```javascript
const express = require('express');
const cors = require('cors');
const fs = require('fs');
const { exec } = require('child_process');
const path = require('path');

const app = express();
// 开启跨域允许前端 (5173端口) 访问
app.use(cors());
// 解析 application/json 格式的请求体
app.use(express.json());

// 核心执行接口
app.post('/api/run', (req, res) => {
  const { code, language } = req.body;
  
  // 目前仅支持 JavaScript 的执行演示
  if (language !== 'javascript') {
    return res.status(400).json({ error: 'Unsupported language' });
  }

  // 1. 生成临时文件路径
  const tempFilePath = path.join(__dirname, 'temp.js');

  // 2. 将前端传来的代码写入临时文件
  fs.writeFileSync(tempFilePath, code);

  // 3. 创建子进程，执行该临时文件
  exec(`node ${tempFilePath}`, (error, stdout, stderr) => {
    // 执行完毕后，无论成功失败，都清理掉临时文件
    if (fs.existsSync(tempFilePath)) {
      fs.unlinkSync(tempFilePath);
    }

    // 如果执行过程中有语法或运行错误
    if (error || stderr) {
      return res.json({ 
        success: false, 
        output: stderr || error.message 
      });
    }

    // 执行成功，返回标准输出
    return res.json({ 
      success: true, 
      output: stdout 
    });
  });
});

const PORT = 3000;
app.listen(PORT, () => {
  console.log(`🚀 后端服务器已启动，监听端口: ${PORT}`);
});
```

### 2. 总指挥部串联核心逻辑 (`App.jsx` 摘录)

在 `App.jsx` 中利用 `fetch` 发起网络请求，并动态修改终端的输出。

JavaScript

```react
// ...省略上部的 import 

function App() {
  const [activeFile, setActiveFile] = useState('index.js');
  // 新增：保存当前编辑器里的代码
  const [currentCode, setCurrentCode] = useState('console.log("Hello, 全栈世界!");');
  // 新增：防抖状态，标记是否正在请求后端
  const [isRunning, setIsRunning] = useState(false);
  // 新增：终端日志状态变更为可变状态
  const [terminalLogs, setTerminalLogs] = useState([
    { id: Date.now(), type: 'info', text: '# 终端已就绪。点击顶部运行按钮执行代码。' }
  ]);

  // 核心函数：触发代码运行
  const handleRunCode = async () => {
    if (!currentCode.trim()) return;
    
    setIsRunning(true);
    // 在终端打印开始执行的提示
    addLog('info', `> 正在执行 ${activeFile} ...`);

    try {
      // 向我们刚才写好的本地 Node.js 服务器发送代码
      const response = await fetch('http://localhost:3000/api/run', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          code: currentCode,
          language: 'javascript'
        })
      });

      const result = await response.json();

      // 根据后端返回的 success 字段决定输出颜色
      if (result.success) {
        addLog('success', result.output || '(无输出结果)');
      } else {
        addLog('error', result.output);
      }
    } catch (err) {
      addLog('error', '❌ 连接后端服务器失败，请检查 Server 是否已启动 (端口3000)。');
    } finally {
      setIsRunning(false);
    }
  };

  // 辅助函数：往终端追加日志
  const addLog = (type, text) => {
    setTerminalLogs(prev => [...prev, { id: Date.now(), type, text }]);
  };

  return (
    // ...布局代码...
    {/* 将方法作为 Props 传递 */}
    <Header activeFile={activeFile} onRunCode={handleRunCode} isRunning={isRunning} />
    
    {/* 编辑器需要 onChange 事件将用户输入同步给 currentCode 状态 */}
    <CodeEditor code={currentCode} setCode={setCurrentCode} />
    
    {/* 终端接收最新的日志数组进行渲染 */}
    <Terminal logs={terminalLogs} />
    // ...
  );
}
```

### 第一部分：引入弹药库 (引入依赖)

```javascript
const express = require('express');
const cors = require('cors');
const fs = require('fs');
const { exec } = require('child_process');
const path = require('path');
```

**💡 思考思路：我需要哪些工具来完成任务？**

- **`express`**: 我们的核心框架。Node.js 原生的 HTTP 模块太繁琐了，Express 帮我们把接收请求、返回响应的流程极度简化。
- **`cors`**: **（极其重要）** 跨域资源共享。因为你的前端在 `localhost:5173`，后端在 `localhost:3000`。浏览器的安全机制（同源策略）默认会拦截这种“串门”请求。引入它就是给前端发放一张“通行证”。
- **`fs` (File System)**: 操作硬盘的模块。我们需要把前端发来的代码字符串，真正变成硬盘上的一个 `.js` 文件。
- **`child_process` (子进程)**: **（核心魔法）** Node.js 允许你在代码里“雇佣”一个临时的系统终端（就像你自己打开 CMD 一样），去帮你敲击 `node xxx.js` 命令。
- **`path`**: 路径向导。不同操作系统（Windows 的 `\` 和 Mac/Linux 的 `/`）路径斜杠不同，用它拼接路径绝对不会出错。

------

### 第二部分：搭建与装配服务器

```javascript
const app = express();
// 开启跨域允许前端访问
app.use(cors());
// 解析 application/json 格式的请求体
app.use(express.json());
```

**💡 思考思路：在开门接客前，服务器需要做什么全局准备？**

- **`app.use(cors())`**: 挂载跨域通行证，拦截器放行。
- **`app.use(express.json())`**: **（数据翻译官）** 前端发过来的是一串干瘪的 JSON 字符串数据。这行代码会自动把字符串解析成好用的 JavaScript 对象，并把它塞进 `req.body` 里，方便我们后面直接调用 `req.body.code`。

### 第三部分：核心业务逻辑 (API 路由)

```javascript

  const { code, language } = req.body;
```

**💡 思考：为什么用 `POST`？**

因为前端发来的代码可能非常长（几百行），`GET` 请求的 URL 长度是有限制的，而 `POST` 请求可以把大量数据藏在隐蔽且宽敞的“请求体 (Body)”中。

```javascript
  if (language !== 'javascript') {
    return res.status(400).json({ error: 'Unsupported language' });
  }
```

**💡 思考：安全与边界防御。**

永远不要相信用户的输入。目前我们的系统只装了 Node 引擎，如果用户强行传了个 `python` 过来，系统会崩溃。提前拦截并返回 400（参数错误）状态码，是后端自我保护的基本素养。

```javascript
  // 1. 生成临时文件路径
  const tempFilePath = path.join(__dirname, 'temp.js');
  // 2. 将前端传来的代码写入临时文件
  fs.writeFileSync(tempFilePath, code);
```

**💡 思考：为什么要把代码写进文件里？不能直接在内存里运行吗？**

虽然 JavaScript 有 `eval()` 函数可以直接运行字符串代码，但在企业开发中，直接 `eval` 用户传来的代码是**极其危险**的（极易引发注入攻击）。

我们采用的是**“沙箱隔离思想”的雏形**：把未知的代码隔离到一个独立的文件 `temp.js` 里，然后用独立的系统进程去运行它。这样即使代码写了死循环，也更容易被管控。`__dirname` 代表当前文件所在的绝对路径。

```javascript
  // 3. 创建子进程，执行该临时文件
  exec(`node ${tempFilePath}`, (error, stdout, stderr) => {
```

**💡 思考：如何让代码真正跑起来？**

`exec` 就相当于你用键盘在终端里敲下了 `node C:\...\temp.js`。它会在后台静默执行，并把执行过程中的三种状态收集回来：

- `error`: 系统级别的崩溃（比如 Node 引擎炸了）。
- `stdout`: 标准输出（也就是代码里 `console.log` 打印出来的正常东西）。
- `stderr`: 标准错误（代码的语法错误、运行报错）。

```javascript
    // 执行完毕后，无论成功失败，都清理掉临时文件
    if (fs.existsSync(tempFilePath)) {
      fs.unlinkSync(tempFilePath);
    }
```

**💡 思考：环保与内存泄漏防范。**

这是很多新手会漏掉的**“擦屁股”**环节。如果你执行完不删除 `temp.js`，如果有 1000 个用户同时点击运行，你的硬盘里就会多出 1000 个垃圾文件。`unlinkSync` 就是强制删除文件的意思。

```javascript
    // 如果执行过程中有语法或运行错误
    if (error || stderr) {
      return res.json({ 
        success: false, 
        output: stderr || error.message 
      });
    }

    // 执行成功，返回标准输出
    return res.json({ 
      success: true, 
      output: stdout 
    });
  });
});
```

**💡 思考：如何给前端提供标准化的“物流反馈”？**

前端是一个比较“笨”的展示层，后端必须给它极其明确的指令。

我们人为规定了一个 JSON 格式契约：

- 必须有一个 `success` 字段（`true` 代表绿色成功，`false` 代表红色失败）。

- 必须有一个 `output` 字段存放具体的文字。

  这样前端拿到结果后，直接写个 `if (res.success)` 就能决定用什么颜色的字打印在终端里了。

### 第四部分：点火启动


**💡 思考：服务器的生命周期。**

指定一个没有被占用的端口（前端默认用 5173，后端常用 3000、8080 等）。`app.listen` 让这个 Node.js 程序进入一个死循环的“监听状态”，时刻竖起耳朵等待前端发来的请求，并在终端打印提示，告诉开发者“引擎已点火”。
