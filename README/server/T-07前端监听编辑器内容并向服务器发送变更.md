# T-07：基于 Socket.io 的代码实时同步

## 1、开发思路

### **第一步：前端“实时汇报”变动**

在 `App.jsx` 总部写一个加强版的函数，它不仅要更新本地状态（让编辑器画面不卡顿），还要顺便拿起 Socket 电话向后端汇报。

### **第二步：后端“认真接听”频道**

后端在 `connection` 内部开辟一个叫 `codeChange` 的专属频道，只要前端喊话，后端就立刻把代码打印在黑框框（终端）里。

------

## 2、实操步骤与代码

### **2.1 前端：总部派发“带对讲机的遥控器” (App.jsx)**

**核心知识点（敲黑板）：** * **状态提升**：必须在总部定义 `handleCodeChange`，这样才能同时控制状态和 Socket。

- **精准派发**：遥控器必须发给 `CodeEditor` 部门，而不是 `Header`。

```jsx
// App.jsx

// 1. 定义加强版函数
const handleCodeChange = (newCode) => {
  // 📸 [总部确认]：收到车间代码
  setCurrentCode(newCode); // 更新本地 State
  socket.emit('codeChange', newCode); // 拨号给后端
};

return (
  // ...
  // 2. 正确派发：把 handleCodeChange 传给 CodeEditor 的 setCode 属性
  <CodeEditor code={currentCode} setCode={handleCodeChange} />
  // ...
)
```

### **2.2 前端：车间按响“报警器” (CodeEditor.jsx)**

**核心知识点：** 利用 Monaco Editor 的 `onChange` 钩子，捕捉用户的每一次击键。

```jsx
// CodeEditor.jsx

export default function CodeEditor({ code, setCode }) {
  return (
    <Editor
      value={code}
      // 这里的 value 就是用户刚打进去的最新的代码
      onChange={(value) => setCode(value ?? '')}
      // ...
    />
  );
}
```

### **2.3 后端：接听并记录来电 (server/index.js)**

**核心知识点：** 监听代码必须写在 `io.on('connection')` 的**大括号内部**，且频道名 `codeChange` 必须与前端完全对齐。

```jsx
// server/index.js

io.on('connection', (socket) => {
  console.log('user connected');

  // 监听前端刚才发出的 'codeChange' 频道
  socket.on('codeChange', (newCode) => {
    // 在后端控制台实时打印
    console.log(`[实时同步] 收到：\n${newCode}`);
  });
});
```

## 3、排雷总结：React 18 “挂断惨案”

这是咱们这次任务中遇到的最大挑战，也是外企面试的高频考点。

### **【现象描述】**

后端终端显示 `user connected` 后立刻显示 `user disconnected`。前端打字时，后端完全没有反应。

### **【错误原因】**

1. **React 18 严格模式 (Strict Mode)**：在开发环境下，React 会为了检查 Bug 故意将组件挂载、卸载、再重新挂载一次。
2. **全局单例冲突**：由于 `socket` 定义在组件外部，第一次卸载时触发了 `useEffect` 的 `socket.disconnect()`。第二次挂载时，原本的 `socket` 实例已经失效，电话被彻底挂断了。

### **【解决方案】**

在单页面应用（SPA）中，保持长连接的稳定性。**注释掉 `useEffect` 里的自动挂断逻辑**，防止 React 误操作导致断线。

JavaScript

```jsx
// App.jsx
useEffect(() => {
  socket.on('connect', () => {
    console.log('拨号成功！');
  });

  return () => {
    // ❌ 注释掉这行，防止 React 严格模式误杀连接
    // socket.disconnect(); 
  };
}, []);
```

## 💡 面试必杀技

**👨‍💼 面试官提问：**

> “在你的 Web IDE 项目中，你提到了使用 Socket.io 实现实时同步。在 React 环境下集成 WebSocket 时，你遇到过什么棘手的工程化问题吗？是怎么解决的？”

**👩‍💻 你的满分回答：**

> “我遇到过一个非常经典的底层生命周期冲突问题，我称它为**‘幽灵断连’**。
>
> 起初在开发 T-07 实时同步时，我习惯性地在 `useEffect` 的 cleanup 函数里写了 `socket.disconnect()`。结果发现后端一直显示连上后立刻断开，前端无论怎么发送数据后端都收不到。
>
> 经过全链路追踪，我发现根本原因在于 **React 18 的严格模式 (Strict Mode)**。为了暴露潜在的副作用，React 在开发环境下会模拟执行一次‘挂载-卸载-重挂载’。这导致我挂载在全局的单例 Socket 被那次模拟卸载给**永久摧毁**了。
>
> 我的解决思路是：在单页面应用（SPA）中，这种负责全局通信的 Socket 实例，它的生命周期应该与整个应用级（App层）一致，而不应该被当做局部组件的副作用去轻易销毁。我移除了那个不合理的 cleanup 逻辑，彻底解决了长连接断连的问题。
>
> 这次踩坑让我深刻认识到，**高级前端开发不能只停留在调用 API 的层面，必须对框架底层的渲染机制和生命周期有透彻的理解**。”

## 💡 进阶考点：展示全栈思维

如果面试官顺势追问：“如果我不仅想把代码发给后端，我还想告诉后端**这段代码是哪个文件里的**，你的 `emit` 该怎么写？”

你要自信地抛出你的工程化思维：

> “Socket.io 的第二个参数天然支持 JSON 对象。当需要传递多个维度的信息时，我不会去发多次 `emit`，而是直接把它们打包成一个对象作为第二个参数发送：
>
> JavaScript
>
> ```
> // 前端打包发送
> socket.emit('codeChange', { 
>   code: newCode, 
>   fileName: activeFile 
> });
> ```
>
> 后端接收时，直接解构这个对象就可以了。这种数据结构的设计，让未来的扩展（比如再加个时间戳、用户名）变得非常容易，且不会破坏原有的代码结构。”

这短短的一行代码背后，其实就是网络通信最经典的 **“事件驱动模型” (Event-Driven Architecture)**。
