# 🚀 进阶重构思路：路径净化（防暴击防作死拦截）

### 1. 标准协同数据流（ should be ）

由于我们引入了 WebSocket 广播机制，后端的清洗和响应不是“点对点”返回，而是“清洗 -> 落地 -> 重新扫描 -> 全员广播”：

```react
[前端输入] (例如: "///src//components/..//Button.jsx")
   │
   ▼ (Socket.emit 'createFile')
[后端拦截] ➔ 1. 安全检查：拦截越界路径 (如 ../)
   │        ➔ 2. 路径净化：将多个斜杠 "///" 缩减为一个 "/"
   ▼
[物理创建] ➔ 在服务器 temp 目录下安全落地
   │
   ▼
[重新扫描] ➔ 后端调用 buildFileTree(tempDir) 重新生成完美的整棵 JSON 树
   │
   ▼ (io.to(roomId).emit 'initCodePackage')
[全员广播] ➔ 房间里的所有人（包括你自己）的 Zustand 仓库被全量覆盖
   │
   ▼
[响应渲染] ➔ 前端组件无感重绘，新文件/文件夹出现在正确的位置
```

## 💻 后端 `index.js` 的全栈工业级重构

为了实现这个防遍历、防手滑的路径清洗，我们需要在后端的 `createFile` 监听器里使用 Node.js 自带的 `path` 核心模块。

请对照你后端的 `backend/index.js`，将 `createFile` 逻辑重构为以下**工业级防御性代码**：

```react
// backend/index.js 内部的 socket.on('createFile') 部分

socket.on('createFile', async (data) => {
  const { roomId, filename, isFolder } = data;

  try {
    // ==========================================
    // 🛡️ 扩展级防线一：路径净化 (Sanitization)
    // ==========================================
    // 1. 利用 path.normalize 自动把 "///src//components/" 缩减为 "src/components"
    // 2. 过滤掉用户通过相对路径作死的行为（如强行解析掉 ..）
    const cleanedFilename = path.normalize(filename)
      .replace(/^(\.\.(\/|\\|$))+/, '') // 剔除开头的 ../
      .replace(/\\/g, '/');            // 统一将 Windows 的反斜杠 \ 转为正斜杠 /

    // 基础边界检查：清洗后如果是空字符串，直接拦截
    if (!cleanedFilename || cleanedFilename === '.' || cleanedFilename === '/') {
      socket.emit('terminalError', `创建失败: 非法的名称或路径`);
      return;
    }

    // 3. 拼接绝对路径：temp/房间ID_清洗后的路径
    const rootFolderName = cleanedFilename.split('/')[0];
    const restPath = cleanedFilename.substring(rootFolderName.length);
    const filePath = path.resolve(tempDir, `${roomId}_${rootFolderName}${restPath}`);

    // ==========================================
    // 🛡️ 扩展级防线二：防越界检查 (Path Traversal Protection)
    // ==========================================
    // 核心安全策略：判断拼接后的绝对路径，是否依然在我们的 tempDir 目录下
    // 如果用户传了 "../../"，resolved 出来的路径会跑到 temp 目录外面去
    if (!filePath.startsWith(tempDir)) {
      socket.emit('terminalError', `安全警告: 越界访问，已被系统拒绝！`);
      console.warn(`[安全警报] 用户尝试越界创建文件: ${filename}`);
      return;
    }

    // ==========================================
    // 🛡️ 第三步：存在性检查
    // ==========================================
    if (fs.existsSync(filePath)) {
      socket.emit('terminalError', `创建失败: ${isFolder ? '文件夹' : '文件'} ${cleanedFilename} 已存在`);
      return;
    }

    // ==========================================
    // 🛠️ 第四步：执行物理创建
    // ==========================================
    if (isFolder) {
      fs.mkdirSync(filePath, { recursive: true });
    } else {
      const dirName = path.dirname(filePath);
      if (!fs.existsSync(dirName)) {
        fs.mkdirSync(dirName, { recursive: true });
      }
      await fs.promises.writeFile(filePath, '', 'utf-8');
    }

    // ==========================================
    // 📡 第五步：全量扫描广播
    // ==========================================
    const codeTree = buildFileTree(tempDir);
    io.to(roomId).emit('initCodePackage', codeTree);

  } catch (err) {
    socket.emit('terminalError', `系统异常: ${err.message}`);
  }
});
```

## 🎓 导师课堂：这一波重构在面试中怎么吹？

在外企前端求职时，简历里写 `Web IDE` 的人很多，但大多数人的项目只是个“玩具”。你在求职时，如果被问到这个项目，完全可以说：

> “在处理文件协同系统的文件创建时，我没有采用浅显的单向数据直发，而是实现了一套**基于 Node.js 核心 Path 模块的防御性全栈链路**。前端通过 WebSocket 发出创建申请，后端控制层拦截并利用 `path.normalize` 和 `path.resolve` 进行动态路径净化。
>
> 同时，为了防止恶意的路径遍历攻击（Path Traversal），我加入了 `startsWith` 的边界沙箱断言，确保所有的物理 IO 操作都被严格锁死在临时的沙箱目录中。确认安全后，再触发目录树重描并利用 Socket.io 进行全房间广播同步。这不仅保证了高并发下协同的实时性，也规避了越界物理写操作带来的系统级安全风险。”

这一套说辞下来，技术深度和全局架构感瞬间就立起来了。

现在你已经把这条链路的安全性封顶了，对于被清洗掉的不规范路径（比如用户打了 `///test`，后端自动帮他修正成了 `test` 落地并展示），你觉得前端需要主动弹出一个 `toast` 提示用户“系统已自动规范路径”来提升交互体验，还是让它静默修正比较好？



然决定要弹，在多人协作的 Web IDE 架构下，我们就面临一个非常有意思的**工业级架构挑战**：

> **挑战**：小明手滑输入了不规范路径，后端帮小明修正了。这时候，应该**只有小明**的屏幕上弹出这个 Toast 警告，房间里正在写代码的**其他队友不应该被这个弹窗打扰**。他们只需要静默看到目录树多了一个干净的文件即可。

为了实现这种“精准点对点”的反馈，我们不能用普通的全局广播（Broadcast），而是要引入一种极具技术含量的面试加分项：**Socket.io Acknowledgements（消息确认机制）**。

## 🚀 什么是 Socket.io Acknowledgements（确认机制）？

平时我们用 Socket 都是单向发消息：前端 `emit` 出去，就不管了。 而 **Acknowledgements** 允许我们在 WebSocket 上模拟出类似 HTTP 的 **“请求-响应”** 链路。前端发消息时可以附带一个回调函数，后端处理完后，可以**直接调用这个函数**把结果单独塞还给触发这个操作的特定前端。

现在，我们把这套高级机制塞进你的项目里！

### 第一步：升级后端控制层（支持回调响应）

打开 `backend/index.js`，找到 `socket.on('createFile')`。我们在参数里加入 `callback`，并在路径发生清洗时，通过 `callback` 悄悄通知该前端。

JavaScript

```react
// backend/index.js 里面的 createFile 逻辑重构

socket.on('createFile', async (data, callback) => { // 💡 注入第二个参数 callback
  const { roomId, filename, isFolder } = data;

  try {
    const cleanedFilename = path.normalize(filename)
      .replace(/^(\.\.(\/|\\|$))+/, '')
      .replace(/\\/g, '/');

    if (!cleanedFilename || cleanedFilename === '.' || cleanedFilename === '/') {
      socket.emit('terminalError', `创建失败: 非法的名称或路径`);
      // 如果前端传了回调，安全返回失败
      if (typeof callback === 'function') callback({ success: false, msg: '非法路径' });
      return;
    }

    const rootFolderName = cleanedFilename.split('/')[0];
    const restPath = cleanedFilename.substring(rootFolderName.length);
    const filePath = path.resolve(tempDir, `${roomId}_${rootFolderName}${restPath}`);

    if (!filePath.startsWith(tempDir)) {
      socket.emit('terminalError', `安全警告: 越界访问！`);
      if (typeof callback === 'function') callback({ success: false, msg: '越界访问' });
      return;
    }

    if (fs.existsSync(filePath)) {
      socket.emit('terminalError', `创建失败: 该名称已存在`);
      if (typeof callback === 'function') callback({ success: false, msg: '同名冲突' });
      return;
    }

    // 执行物理创建
    if (isFolder) {
      fs.mkdirSync(filePath, { recursive: true });
    } else {
      const dirName = path.dirname(filePath);
      if (!fs.existsSync(dirName)) fs.mkdirSync(dirName, { recursive: true });
      await fs.promises.writeFile(filePath, '', 'utf-8');
    }

    // 全量扫描广播给所有人（更新目录树视图）
    const codeTree = buildFileTree(tempDir);
    io.to(roomId).emit('initCodePackage', codeTree);

    // ==========================================
    // 💡 核心魔法：只点对点回应当前的操作者
    // ==========================================
    if (typeof callback === 'function') {
      // 检查路径是否被清洗过（原输入和清洗后的输入不一致）
      const isSanitized = filename !== cleanedFilename;
      
      callback({
        success: true,
        isSanitized,          // 告诉前端：我帮你洗过路径了
        original: filename,   // 用户原本输入的作死路径
        cleaned: cleanedFilename // 净化后的标准路径
      });
    }

  } catch (err) {
    socket.emit('terminalError', `系统异常: ${err.message}`);
    if (typeof callback === 'function') callback({ success: false, msg: err.message });
  }
});
```

### 第二步：升级前端调度层（接收确认并弹出 Toast）

现在打开 `frontend/src/App.jsx`，重构 `handleCreateFile` 函数。我们利用你已经安装好的 `react-hot-toast`，对路径清洗进行精美的视觉反馈。

JavaScript

```react
// frontend/src/App.jsx 里面的 handleCreateFile 函数

const handleCreateFile = ({ path, isFolder }) => {
  if (currentSocket) {
    // 💡 发送事件时，第 3 个参数传入一个回调函数，用来接收后端的点对点确认
    currentSocket.emit('createFile', { roomId, filename: path, isFolder }, (response) => {
      
      // 如果后端反馈成功，且触发了路径净化
      if (response && response.success && response.isSanitized) {
        
        // 弹出极具外企人文关怀的优雅提示
        toast(
          (t) => (
            <div className="text-xs text-gray-600">
              <p className="font-bold text-amber-500 mb-0.5">⚠️ 系统已自动规范路径</p>
              <p>由于检测到不规范输入，已将：</p>
              <code className="bg-gray-100 px-1 py-0.5 rounded text-rose-500 line-through block my-0.5 truncate">{response.original}</code>
              <p>自动修正为标准路径：</p>
              <code className="bg-gray-50 px-1 py-0.5 rounded text-emerald-600 block my-0.5 font-medium truncate">{response.cleaned}</code>
            </div>
          ),
          {
            duration: 5000, // 停留5秒，给用户足够的阅读时间
            icon: '⚙️',
            style: {
              border: '1px solid #f59e0b',
              padding: '12px',
              background: '#fff',
            },
          }
        );
      }
    });
  }
};
```
