### 🌐 架构笔记：前后端交互原理与全栈流转流程

#### 1. 📝 核心概念：它们到底是怎么交流的？

前端和后端就像是餐厅里的“前厅服务员”和“后厨大厨”。

- **前端（客户端）**：负责展示菜单、收集客人的点单（收集代码），但它自己不会做菜。
- **后端（服务端）**：待在安全的后厨，拥有各种厨具（Node.js 底层环境、文件系统），负责把菜做熟（执行代码）。
- **交互的桥梁：HTTP 协议与 JSON**
  - 它们之间不能直接说话，必须通过 **HTTP 协议**（相当于传菜员）来传递信息。
  - 传递的信息必须使用双方都能看懂的统一格式，这就是 **JSON**（一种轻量级的文本数据交换格式）。前端把 JavaScript 对象变成 JSON 文本发过去，后端收到 JSON 文本后再还原成自己能看懂的对象，反之亦然。

#### 2. 🔄 核心执行流程（以 Web IDE 点击“运行”为例）

一次完整的交互，被称为一次 **Request/Response (请求/响应) 生命周期**。下面是逐帧拆解：

**📍 阶段一：前端发起请求 (打包发件)**

1. **收集数据**：在 `App.jsx` 中，当用户点击运行按钮，前端的 `handleRunCode` 函数被触发。它把 `currentCode`（代码字符串）和 `language` 组合成一个普通的 JS 对象。
2. **序列化 (JSON 转换)**：因为网络线路上只能传输纯文本，不能传输内存里的对象。前端必须调用 `JSON.stringify()`，把数据打包成规范的 JSON 文本串。
3. **发送请求 (Fetch)**：前端调用 `fetch('http://localhost:3000/api/run', { method: 'POST', body: ... })`。这相当于把打包好的 JSON 文本交给了浏览器底层的网络模块，目标地址是后端的 API 接口。

**📍 阶段二：网络传输 (跨越结界)**

- 包裹带着前端的数据，通过本机的网络栈发送。这里可能会受到跨域安全策略（CORS）的拦截，这也是为什么我们必须在后端的 `server/index.js` 里写上 `app.use(cors())`，相当于给后厨发了一张通行证，允许接收来自前厅 5173 端口的包裹。

**📍 阶段三：后端接收与处理 (拆包干活)**

1. **接收与反序列化**：Node.js 后端的 `app.post` 路由拦截到了包裹。中间件 `express.json()` 就像个自动拆包机，把 JSON 文本重新变回 JavaScript 对象，并挂载到 `req.body` 上。
2. **核心业务逻辑**：后端从 `req.body.code` 拿出代码。因为底层系统只认文件不认内存字符串，后端通过 `fs.writeFileSync` 把它写成一个真实的临时 `.js` 文件。
3. **系统调用**：后端调用 `child_process.exec`，唤醒系统底层的 Node 引擎去执行这个临时文件，并监听执行的输出结果（`stdout` 标准输出，或 `stderr` 报错信息）。

**📍 阶段四：后端返回响应 (组装回信)**

- 后端清理掉临时文件后，把执行结果再次打包成对象 `{ success: true, output: "..." }`，通过 `res.json()` 发送出去。这个方法会自动把对象变成 JSON 文本，顺着 HTTP 协议的原路寄回给前端。

**📍 阶段五：前端渲染结果 (拆信展示)**

1. **接收回信**：前端 `App.jsx` 一直在耐心等待 (`await response`)。收到回信后，执行 `await response.json()`，把 JSON 文本解析回 JS 对象。
2. **状态更新与驱动视图**：前端根据对象的 `success` 字段是 true 还是 false，调用 `addLog` 更新 React 的状态。状态的改变最终驱动 `Terminal.jsx` 在屏幕上画出绿色或红色的文字。整个周期完美闭环。

#### 3. 🧠 为什么要这样设计？(架构思考思路)

- **绝对的解耦 (Decoupling)**：前端只管“长得好看和收集操作”，后端只管“处理数据和保证安全”。哪怕哪天你不想用 Node.js 了，想换成 Java Spring Boot 来写后端，只要中间交互的 **JSON 格式** 和 **URL 地址** 不变，前端的代码一行都不用改！
- **异步等待与防御性编程**：因为网络传输是不稳定的（就像平时配置 npm 镜像或系统代理时偶尔遇到的网络卡顿一样），请求随时可能延迟或失败。所以前端必须使用 `async/await` 让代码在此等候，同时用 `try...catch` 捕捉网络断开的严重错误。在此期间，通过 `isRunning` 状态锁住“运行”按钮，防止用户因为心急而疯狂连击，导致发送几百个请求把后端服务器撑爆。
