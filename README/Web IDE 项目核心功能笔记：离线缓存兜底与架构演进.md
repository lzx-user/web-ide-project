# Web IDE 项目核心功能笔记：离线缓存兜底与架构演进

## 1. 需求背景与核心目标
在 Web IDE 的多人协同场景中，网络波动或用户误关网页（断网、手滑刷新）会导致未保存的代码彻底丢失。
**核心目标 (Phase 2)**：实现一套“离线缓存兜底”机制，确保在任何意外断开连接的情况下，用户的代码草稿都不会丢失，下次进入房间可无缝恢复。

---

## 2. 技术选型与架构演进（Trade-off）

### 方案 A：Yjs + IndexedDB（初期尝试）
* **思路**：使用 CRDT（无冲突复制数据类型）算法库 `Yjs` 配合 `y-indexeddb` 实现底层的状态同步和本地 IndexedDB 离线存储。
* **痛点**：React 的生命周期（组件挂载/卸载）与 Monaco Editor 的异步挂载存在极其复杂的竞态关系。例如使用 `useRef` 捕获编辑器实例时，首次渲染为 `null` 导致 Yjs 无法正确绑定模型，引发了各种同步冲突和“正在恢复草稿”状态卡死的问题。
* **架构决策 (Trade-off)**：在有限的交付排期内，为了保证项目的高可用性和稳定性，果断进行**技术降级**，采用方案 B。这种“为了按时交付业务价值而舍弃高风险技术复杂度”的取舍，体现了成熟的外企敏捷工程化思维。

### 方案 B：LocalStorage + 防抖实时缓存（最终方案）
* **核心思路**：
  1. **实时写入**：拦截用户的键盘输入事件，在向服务器发送 socket 广播的同时，利用 `localStorage.setItem` 将当前代码实时存入本地硬盘，使用 `draft-[roomId]-[filename]` 作为独立命名空间，防止多房间文件串台。
  2. **挂载读取**：在用户切换文件或首次拉取服务器代码包时，利用 `localStorage.getItem` 提取本地草稿，并与服务器下发的代码进行“PK”，优先展示本地草稿。

---

## 3. 核心踩坑与 Bug 解决实录

在重构和实现方案 B 的过程中，遇到了四个极具代表性的高频 Bug，现复盘如下：

### 🐛 Bug 1：空房间总是默认显示幽灵 `index.js` 路径
* **现象描述**：输入全新房间号进入后，左侧资源管理器为空，但顶部面包屑和编辑器均默认挂载着 `index.js`。
* **根本原因分析**：
  1. **后端拦截过度**：Node.js 后端在处理加入房间逻辑时，加了 `if (Object.keys(codePackage).length > 0)` 判断。如果是空房间，后端直接不发送 `initCodePackage` 广播。
  2. **前端状态无更新**：Zustand 全局状态 `activeFile` 初始值写死了 `'index.js'`，因为没收到后端的空包广播，前端无法触发清空逻辑。
* **解决思路**：
  * **后端**：移除 `if` 判断，无条件向前端发射代码包（哪怕是空对象 `{}`）。
  * **前端**：在 `App.jsx` 监听到 `initCodePackage` 时，若发现包为空，主动执行 `setActiveFile('')` 和 `setCurrentCode('')`；同时将 Zustand 中的默认值改为 `''`。

### 🐛 Bug 2：断网刷新后，代码完全无法恢复
* **现象描述**：断网输入代码后刷新页面，再重新连接进入，代码并没有从 LocalStorage 恢复。
* **根本原因分析**：
  1. **“存”的缺失**：在 `handleCodeChange` 中，只更新了 React 状态并发送 Socket，忘记写 `localStorage.setItem`，导致草稿根本没落盘。
  2. **“读”的错位**：在调用 `monaco.editor.createModel` 时，传入了 4 个参数（空字符串、草稿代码、语言、URI），导致真正的离线草稿被 Monaco 当成了“语言”配置给忽略了。
  3. **组件通信断层**：瘦身后的 `CodeEditor.jsx` 变成了只读组件，但父组件忘记将 `code`、`onChange` 和 `onMount` 方法以 props 的形式传给它，导致完全失联。
* **解决思路**：
  * 在 `handleCodeChange` 中补齐 `localStorage.setItem(cacheKey, newCode)` 核心代码。
  * 将 `createModel` 严格收敛为 3 个标准参数：`(最终代码, 'javascript', URI)`。
  * 补全父子组件的 Props 传递链路，确保实例被正确传回 `App.jsx`。

### 🐛 Bug 3：刷新页面后，本地草稿被“空文件”暴力覆盖
* **现象描述**：代码成功存入了 LocalStorage，但刷新页面重新进入时，草稿只闪了一下就消失了。
* **根本原因分析**：没有处理好“单点真相（Single Source of Truth）”的优先级。前端在收到后端的 `initCodePackage` 时，盲目信任了服务器下发的数据。因为代码未点保存，服务器发来的是空内容，直接覆盖了本地刚从 LocalStorage 里读出来的草稿。
* **解决思路 (Draft PK 机制)**：
  * 收到服务器下发的代码 `code` 时，不直接更新 Model。
  * 先执行 `const draftCode = localStorage.getItem(...)`。
  * 编写 PK 逻辑：`const finalCode = draftCode !== null ? draftCode : code;`，确保本地离线草稿拥有最高优先级。

### 🐛 Bug 4：新建一个文件，出现三个影分身
* **现象描述**：点击新建文件输入 `test.js` 后，文件树里一下子蹦出来 3 个 `test.js`。
* **根本原因分析**：代码合并时冗余。`handleCreateFile` 函数中不仅 `socket.emit` 告诉后端去创建，还手动使用 `setFileList` 在本地强行塞入一个对象。由于后端会把创建成功的消息再广播回来（`fileCreated` 事件），导致前端重复添加。
* **解决思路**：遵循“状态以服务器统一调度为准”的原则。把前端手动操作本地文件数组的逻辑全部删掉，`handleCreateFile` 只负责“向后端发送指令”，前端统一监听后端的广播来更新 UI。

## 4. 面试亮点提取（针对外企研发岗）

当面试官问到项目难点时，可按此逻辑阐述：
> “在这个项目中，为了保证弱网和断网情况下的用户代码安全，我设计了离线兜底机制。
>
> 初期我尝试引入 `Yjs` 实现基于 CRDT 的离线协作，但在集成过程中发现了 React 生命周期与 Monaco 异步挂载极易产生竞态冲突（Race Condition）。考虑到项目敏捷交付的 Time-to-Market 要求，我果断做出了 **技术降级（Trade-off）**，转而采用更加轻量稳定的 `localStorage` + 防抖机制。
>
> 在这个过程中，我重点解决了两个工程化难题：
> 1. **状态一致性**：引入了本地草稿与服务端拉取数据的优先级 PK 机制，防止刷新时服务端旧数据覆盖本地新草稿。
> 2. **单点真相 (Single Source of Truth)**：重构了文件创建的事件流，移除前端本地乐观更新，强制统一监听服务端的广播结果，彻底解决了高并发下的状态撕裂（影分身）问题。
>
> 最终仅用不到 20 行核心代码，就稳定覆盖了 100% 的断网丢失痛点，系统复杂度大幅降低。”
