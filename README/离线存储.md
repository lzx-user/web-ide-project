# Web IDE 项目核心功能笔记：离线缓存兜底与架构演进

## 1. 需求背景
在 Web IDE 多人协作场景下，网络波动、浏览器崩溃或用户意外刷新页面（断网/手滑）会导致未保存的代码草稿彻底丢失。为了提升用户体验（UX）并展现工程严谨性，我们需要实现一套**离线缓存兜底机制**。

---

## 2. 核心实现思路 (Implementation Logic)

这套方案的核心在于：**“本地实时镜像，挂载优先对齐”**。

### A. 数据存储设计 (Storage Schema)
* **存储介质**：选用 `LocalStorage`。
* **Key 的设计**：采用命名空间隔离，防止多房间数据冲突。
  * 格式：`draft-${roomId}-${filename}`
  * 示例：`draft-999-index.js`
* **Value**：存储纯文本字符串（编辑器当前的最新代码内容）。

### B. 写入策略：实时镜像 (Push / Write Strategy)
* **触发时机**：在编辑器 `onChange` 回调中执行。
* **逻辑流**：每当用户敲击键盘，代码进入 React 状态的同时，同步执行 `localStorage.setItem`。
* **防抖优化**：虽然 LocalStorage 写入很快，但在极高频率输入下，建议配合防抖（Debounce）逻辑，以减少不必要的 I/O 开销。

### C. 读取策略：两阶段恢复 (Pull / Read Strategy)
1. **初始同步阶段 (Init Sync)**：
   * 当用户进入房间收到后端发来的 `initCodePackage`（历史代码包）时，不直接覆盖编辑器。
   * 逻辑：循环遍历包内文件，先去 LocalStorage 查找是否有对应的 `draft` 缓存。
   * **优先级策略 (PK Logic)**：`const finalCode = localDraft !== null ? localDraft : serverCode;`（本地草稿优先级最高）。
2. **文件切换阶段 (File Switching)**：
   * 在切换 `activeFile` 的 `useEffect` 逻辑中，如果该文件尚未创建 Monaco Model，则优先从本地缓存读取内容进行初始化。

## 3. 技术选型演进 (Trade-off)

### 初期尝试：Yjs + IndexedDB
* **思路**：利用 CRDT 算法和 `y-indexeddb` 实现离线协作。
* **挑战**：发现 Monaco Editor 的异步挂载与 React 生命周期存在严重的竞态冲突（Race Condition），导致 Ref 引用为 `null` 或协同实例绑定失败。
* **工程决策**：本着**敏捷开发（Agile）**和**按时交付（Time-to-Market）**的原则，果断切换到更稳健、更易维护的 LocalStorage 方案。这种能根据项目周期进行“技术降级”的能力，是成熟开发者的体现。

## 4. 核心 Bug 复盘与解决 (Post-Mortem)

### 🐛 Bug 1：空房间总是显示幽灵 `index.js` 路径
* **原因**：Zustand 状态 `activeFile` 初始值写死为 `'index.js'`，且后端在房间为空时不发送初始化广播，导致前端无法重置状态。
* **解决**：后端取消 `if` 限制，强制发送代码包（哪怕为空）；前端在接收到空包时主动清空 `activeFile`。

### 🐛 Bug 2：文件创建产生“影分身” (Duplicate Files)
* **原因**：前端既手动乐观更新了 `fileList`，又监听了后端的创建广播，导致重复渲染。
* **解决**：遵循 **“单点真理 (Single Source of Truth)”** 原则，前端不再手动修改文件列表，统一由后端广播驱动 UI 更新。

### 🐛 Bug 3：离线代码被服务器空内容覆盖
* **原因**：进入房间时，前端盲目信任服务器下发的（未保存的）空文件。
* **解决**：引入 **PK 机制**，在创建 Model 前强制检查本地是否有未提交的草稿，并以本地草稿为准。
