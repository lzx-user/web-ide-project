### 🧱 知识沙盘第五站：Node-pty 与 沙箱执行引擎

#### 1. 它们的核心作用是什么？

在你的 Web IDE 里，这其实是两条完全不同的底层执行路线，它们分工极其明确：

- **Node-pty（交互式伪终端）**：负责底层 Shell 交互。它在后端真正克隆（Fork）出了一个操作系统的控制台进程（Bash 或 PowerShell）。用户在前端 `Xterm.js` 里敲击的每一个字母，都会被无缝传递给这个后端进程；它吐出的结果（哪怕是带有红绿颜色控制符的 ANSI 乱码），也会原封不动流回前端。
- **`child_process.exec`（原子化沙箱执行）**：负责安全地运行用户写好的那一份 JS 代码。它在主服务之外开辟了一个独立的“安全屋（子进程）”，并戴上了“5秒钟倒计时手铐”。只负责执行、拿结果、销毁，绝不拖泥带水。

#### 2. 为什么要分开设计？（不用的痛点是什么？）

在外企后端/全栈面试中，**进程安全性**是绝对的红线。

- **如果只用 `eval()` 运行用户代码的痛点**：这是绝对的禁忌！如果在 Node.js 主进程里直接执行用户传来的字符串，只要用户写一句 `while(true) {}`，你的整个服务器 CPU 瞬间拉满，主线程当场卡死，所有房间的队友全部掉线。如果用户写一句 `fs.rmdirSync('/', {recursive: true})`，你的服务器硬盘直接被清空（删库跑路）。
- **如果只用普通的 `exec` 来做终端的痛点**：普通的 `exec` 是**缓冲模式（Buffered）**。它必须等脚本全部跑完，才会一次性把结果返回给你。如果你用它来执行 `npm install` 这种需要跑几分钟的命令，前端就只能死等，看不到进度条流转。更致命的是，普通的 `exec` **不支持交互**，你没法输入 `Y/N` 来确认操作。
- **所以，完美的架构是**：用 `node-pty` 做长效的、流式的、交互式终端；用 `child_process.exec` 配合沙箱做瞬时的、隔离的、安全的纯代码计算。

#### 3. 结合你的源码进行深度拆解

你的后端代码在这里写得极其老练，处处都是防御性编程（Defensive Programming）的影子：

##### 阶段 A：PTY 进程的创建与生命周期管理

- **【源码出处】** `backend/src/pty/PtyManager.js`

- **【工程落地】** 你将每个终端封装成了一个独立的 Class 实例。当用户进入房间：

  ```js
  // 1. 在指定的虚拟工作区（cwd）fork 出真实的操作系统终端
  this.ptyProcess = pty.spawn(shell, args, { cwd: workspaceDir, cols: 80, rows: 30 });
  
  // 2. 管线对接：把终端进程吐出的流，用 socket 发给前端
  this.ptyProcess.on('data', (data) => { this.socket.emit('terminal-out', data); });
  
  // 3. 防僵尸进程销毁：用户断开时，连根拔起杀掉整个进程树
  destroy() { if (this.ptyProcess) this.ptyProcess.kill(); }
  ```

##### 阶段 B：安全沙箱的“三重锁”防御

- **【源码出处】** `backend/index.js` 中的 `executeCode` 事件

- **【工程落地】** 当用户点击【运行】按钮时，你并没有直接跑他的代码，而是上了三把锁：

  - **第一重锁：`require` 关键字拦截（API 封杀）**。你在用户代码外面包了一层马甲，强行重写了 `require`。如果用户企图引入 `fs`、`os`、`child_process` 等敏感模块，直接抛出异常，斩断了黑客操控服务器硬盘的可能。

  - **第二重锁：独立的 `exec` 子进程隔离**。你生成了一个随机的 `stream_uuid.js`，让操作系统另外开一个进程去跑它，主 Node.js 服务绝对不碰这段危险代码。

  - **第三重锁：`timeout: 5000`（死循环熔断）**。这是最亮眼的细节：

    JavaScript

    ```js
    exec(`node ${fileName}`, { cwd: roomDir, timeout: 5000 }, (error, stdout, stderr) => { ... })
    ```

    哪怕代码里写了死循环，只要超过 5 秒，操作系统底层就会发送 `SIGKILL` 信号强制干掉这个子进程。主服务器依旧坚如磐石。

#### 4. 面试死磕防线与防御话术

面对这种全栈级别的架构，高级主考官一定会用极端情况来试探你的底层认知：

**⚔️ 追问一：“为什么不用 `node-pty` 去执行用户点击【运行】发来的那段 JS 代码？非要另起一个 `child_process.exec`？这不是杀鸡用牛刀吗？”**

> **🛡️ Senior 架构师防御话术：** “两者的定位完全不同。`node-pty` 维护的是一个有状态的、持续交互的终端会话（Session）。如果用它来执行代码，一旦用户的代码抛出致命异常或者引发阻塞，整个 PTY 进程就会被污染甚至卡死，用户必须手动刷新页面才能恢复终端功能。 而点击【运行】代码属于无状态的纯计算逻辑，我采用 `child_process.exec`，是因为它可以精准配置 `timeout: 5000` 来做系统级的死循环熔断，并且执行完毕后进程立刻释放，资源回收率极高，不会对维持文件系统交互的 PTY 主管线造成任何干扰。”

**⚔️ 追问二：“你现在的沙箱只重写了 `require`，如果黑客使用类似 `process.exit()` 或者读取全局环境变量 `process.env.JWT_SECRET`，你的系统防得住吗？”**

> **🛡️ Senior 架构师防御话术（展现进阶视野）：** “这正是我们下一阶段优化的重点。目前拦截 `require` 属于应用层防御，能防住基础的文件越界，但防不住针对全局变量（如 `process`）的恶意调用。 若要实现真正的企业级隔离，下一步我会在架构中引入 **Docker 容器化动态调度**，或者使用 Node.js 内置的 **`vm2` (或 `node:vm`) 核心模块** 构建真正的 V8 隔离上下文沙箱，从底层剥离 `process` 和网络访问权限，将 Web IDE 的安全性提升到公有云级别。”

为了让你亲自见证这“三重防御锁”是如何在服务器被恶意攻击时挽救主线程的，我为你搭建了这个 **【Node.js 进程隔离与安全沙箱执行模拟器】**。

在这个沙盘里，你不仅能体验流式终端的输出，更可以扮演“黑客”，亲自在输入框里写入 `while(true)` 死循环或者 `require('fs')` 敏感操作，然后点击运行。你可以亲眼看到子进程是如何被强行 Kill 掉，而代表主服务的呼吸灯依然稳稳闪烁的震撼场面。
