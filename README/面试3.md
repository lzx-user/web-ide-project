### 第一站：项目是什么，解决什么问题

这个项目叫 **Web IDE 协作空间**，核心功能是：

多个人可以同时在浏览器里编辑同一份代码，实时看到对方的光标和内容，还能在线运行代码、使用终端。本质上是一个**简化版的 CodeSandbox 或 StackBlitz**。

用一句话概括技术挑战：**怎么让多人同时编辑同一个文件，且不互相覆盖？**

------

### 第二站：整体架构

```
前端 (Vercel)          后端 (Render)
  React                  Express
  Monaco Editor     →    Socket.io  (控制面：房间、文件、运行)
  Yjs + y-websocket →    Yjs Server (数据面：代码同步)
  Zustand                node-pty   (真实终端)
  xterm.js               JWT        (身份验证)
```

**两条通道，职责完全分离：**

Socket.io 是**控制面**，负责：加入房间、文件树更新、运行代码、终端 I/O 这些"指令类"消息。

Yjs WebSocket 是**数据面**，负责：代码内容的实时同步，用的是 CRDT 算法，天生支持多人并发编辑不冲突。

这两条通道都跑在**同一个端口（3000）**上，通过 `server.on('upgrade')` 事件做路由分流——URL 以 `/yjs/` 开头的交给 Yjs 处理，其他的交给 Socket.io。

### 第三站：用户进入房间的完整流程

```
1. 用户输入昵称和房间号，点击进入
2. 前端发 POST /api/join 到后端
3. 后端用 JWT 签发 Token，返回给前端
4. 前端把 Token 存进 localStorage
5. 前端用 Token 建立 Socket.io 连接
6. 后端 Socket.io 中间件验证 Token，通过才放行
7. 连接建立后，后端立刻 emit initCodePackage，把文件树发给前端
8. 同时前端初始化 Yjs，建立 WebSocket 数据通道
9. 用户点击文件，前端创建 Monaco Model，绑定 Yjs 共享文本
10. 开始协同编辑
```

### 第四站：协同编辑核心——Yjs 怎么工作的

**关键概念：CRDT（无冲突复制数据类型）**

传统解决方案是 OT（操作变换），Google Docs 用的就是这个，需要一个中央服务器来仲裁所有操作的顺序，实现复杂。

Yjs 用的是 CRDT，每个客户端都有一份完整的文档副本，任何本地修改都会被打上时间戳和客户端 ID，合并时按数学规则自动解决冲突，**不需要中央仲裁，天生支持离线后重连同步**。

**在项目里的具体实现：**

js

```js
// 每个房间一个 Y.Doc，相当于一个共享数据库
const ydoc = new Y.Doc();

// 每个文件对应 ydoc 里的一个共享文本
const ytext = ydoc.getText('index.js');

// 把 Monaco 编辑器和 ytext 绑在一起（这就是"胶水"）
const binding = new MonacoBinding(ytext, monacoModel, new Set([editor]), provider.awareness);
```

用户在 Monaco 里打字 → MonacoBinding 把操作同步给 ytext → ytext 通过 WebsocketProvider 发给服务器 → 服务器广播给同房间其他人 → 其他人的 ytext 更新 → MonacoBinding 把变化写回他们的编辑器。

**为什么切换文件要先 `binding.destroy()`？**

因为每个文件对应一个不同的 ytext，如果不撕掉旧的绑定，你在 `index.js` 里打字会同时同步到 `app.js` 里，数据串台。

### 第五站：文件切换的完整逻辑

这是项目里最精妙的部分，在 `App.jsx` 的 `useEffect` 里：

```
1. 保存旧文件的光标位置（viewState）到 fileCacheMap
2. 销毁旧文件的 Yjs 绑定（bindingRef.current.destroy()）
3. 查 fileCacheMap 有没有新文件的 Monaco Model
   - 有：直接复用
   - 没有：用 monaco.editor.createModel() 创建新的
4. editor.setModel() 切换到新模型
5. 恢复新文件的光标位置
6. 为新文件创建 Yjs 绑定
```

**为什么用 fileCacheMap 缓存 Model？**

Monaco Editor 的 Model 创建是有成本的，更重要的是每个 Model 记录了光标位置、滚动位置、撤销历史。如果每次切换文件都销毁重建，用户体验会很差——就像 VSCode 切换文件还能回到上次光标位置，靠的就是这个缓存。

### 第六站：终端是怎么实现的

用的是 `node-pty`，它能在 Node.js 里创建一个真实的伪终端进程（bash 或 PowerShell），不是模拟的，是真的 Shell。

```
用户在 xterm.js 里按键
→ 前端 emit terminal-in 事件，把按键数据发给后端
→ 后端 PtyManager 把数据 write 进 ptyProcess
→ bash 执行，输出结果
→ 后端监听 ptyProcess 的 data 事件，emit terminal-out 发给前端
→ 前端 xterm.js 的 term.write() 把输出渲染到终端
```

**每个用户有独立的 PTY 进程**，互不干扰。用户断开连接时，`PtyManager.destroy()` 会 kill 掉对应的进程，防止僵尸进程堆积。

终端的工作目录被锁死在 `/temp/房间号/` 目录下，用户跑不出这个沙箱。

### 第七站：代码运行是怎么实现的

和终端分开，用 `child_process.exec` 独立执行：

```
1. 生成随机文件名 stream_uuid.js
2. 把用户代码 + 沙箱防御代码包装后写入这个文件
3. exec('node stream_uuid.js') 执行
4. 拿到 stdout 和 stderr 推给前端 Output 面板
5. 无论成功失败，立刻 unlink 删掉这个临时文件
```

**沙箱防御：** 在用户代码外面包了一层，把 `require` 函数劫持，凡是尝试 `require('fs')` 或 `require('child_process')` 这些危险模块，直接抛错。

**超时熔断：** `exec` 配置了 `timeout: 5000`，5 秒内没跑完强制 kill，防止用户写死循环把服务器 CPU 跑满。

### 第八站：状态管理——Zustand

全局状态放在 `useIDEStore`，包括：`isJoined`、`roomId`、`socket`、`activeFile`、`fileList`、`outputLogs` 等。

选 Zustand 而不是 Redux 的原因：API 极简，不需要 action/reducer 那套繁琐模板，适合中小型项目。

**关键设计：** Socket 实例放在 Zustand 里（`socket: null`），而不是用 React state。这样跨组件访问 socket 不需要层层 prop drilling，Header、Sidebar、XTerminal 都能直接从 store 里拿到 socket 发消息。

------

### 第九站：持久化——刷新不丢失登录状态

登录后把 Token、roomId、isJoined 存进 `localStorage`。

刷新页面时，`App.jsx` 的初始化 `useEffect` 检查 localStorage，如果有效就自动恢复状态、重建 Socket 连接，用户感知不到刷新。

Yjs 用了 `y-indexeddb`，把代码内容缓存在浏览器的 IndexedDB 里，即使断网也能继续编辑，重连后自动把本地改动和服务器同步合并。

### 第十站：部署架构

```
前端：Vercel（自动 CI/CD，push 自动部署）
后端：Render 免费实例（不活跃 15 分钟后休眠）
通信：前端 → Render 的单端口服务
```

针对 Render 冷启动问题（首次请求需要 30~50 秒唤醒），前端做了专门处理：Socket 超过 3 秒没连上就显示"正在唤醒"遮罩，防止用户以为系统挂了。
