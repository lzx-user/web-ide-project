### 如何打破对 AI 的依赖？(刻意练习法)

为了让你真正把这套代码变成你自己的，我建议你马上在项目中尝试以下三个步骤：

- **第一步：先写注释，再填空 (Rubber Duck 编程)** 把你刚才发给我的那 4 行注释，原封不动地贴到你的 `App.jsx` 里。然后强迫自己不要看我之前发的代码，只看着注释，尝试用你记忆中的 React 知识去写。哪怕写得语法全错也没关系。
- **第二步：只查官方文档，不问 AI** 当你卡在“怎么接收 Socket 消息”时，克制住问我的冲动。去打开我之前发给你的官方文档，搜索 `socket.on` 的用法。把找 API 的过程变成自己的肌肉记忆。
- **第三步：故意破坏代码 (Chaos Monkey)** 把运行成功的代码故意改错。比如，把 `terminalLogs` 的追加逻辑 `prev => [...prev, newMsg]` 改成 `prev => [newMsg]`，看看终端会发生什么。通过观察错误，你会彻底理解代码为什么要这么写。

你现在已经完全拿捏了前端的业务流转逻辑，这才是最难啃的骨头。代码只是一层皮，多敲几遍自然就有了。

既然你有了这样的觉悟，对于接下来的 **T-11 后端部分（也就是真正涉及系统安全、文件读写和子进程的深水区）**，我们换一种玩法：我不直接给你写好代码了，我只给你流程图和必须用到的 Node.js 原生 API 名称，你根据思路自己在这个 Web IDE 的沙箱里试着手敲出来，遇到具体的报错我们再一起 debug，你愿意接受这个挑战吗？
