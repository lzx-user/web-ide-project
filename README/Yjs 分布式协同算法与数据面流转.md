### 🧱 知识沙盘第四站：Yjs 分布式协同引擎 (CRDT)

#### 1. 它的核心作用是什么？

在 Web IDE 中，Yjs 负责解决一个极其硬核的分布式难题：**“当用户 A 和用户 B 在同一秒钟，对同一行代码的同一个位置敲下不同的字母时，如何保证两人的屏幕最终显示完全一致的代码，且谁的代码都不会凭空消失？”** 它将普通的纯文本字符串，变成了一个底层带有时间戳、客户端 ID 和逻辑时钟的**单向数学链表（CRDT 模型）**，实现了极其精准的“字符级别无冲突合并”。

#### 2. 为什么要用它？（不用的痛点是什么？）

在外企面试中，关于协同方案的技术选型（Trade-offs）是必考题。如果不使用 Yjs（CRDT），通常只有两条路，但都是死胡同：

- **死胡同 A：Naive 全量覆盖（Last-Write-Wins）。**
  - **做法**：A 打字，把整个文件的代码发给后端，覆盖数据库；B 打字，也全量覆盖。
  - **痛点**：这是灾难。A 辛辛苦苦写了 10 行代码，B 只要在另外一行敲了一个回车，B 的旧全量数据发过去，A 写的 10 行代码瞬间被抹除。这根本没法协同。
- **死胡同 B：OT 算法（Operational Transformation，如 Google Docs）。**
  - **做法**：基于“操作转换”。服务器作为唯一的“中央裁判”，接收所有的插入/删除坐标，进行复杂的矩阵换算。
  - **痛点**：服务器计算压力极大（高并发下容易 CPU 熔断），并且**天然不支持离线合并**。一旦断网，本地代码和服务器代码的坐标轴完全偏离，重连时极易发生死锁或崩溃。

**Yjs (CRDT) 的降维打击**： 它去掉了“中央裁判”。任何操作都在本地立刻生效（零延迟体验）。每个字符都有全球唯一的“身份证 `(ClientID, Clock)`”。哪怕断网 3 天，重连的一瞬间，系统也能根据数学定理，像拉链一样把两端的字符完美、确定性地咬合在一起，绝不丢失任何一次击键。

#### 3. 结合你的源码进行深度拆解

在你的项目中，协同层的代码写得非常教科书，完美融合了网络、数据库与编辑器 UI：

##### 阶段 A：建立数据面专线与离线堡垒

- **【源码出处】** `frontend/src/hooks/useWorkspaceSocket.js`

- **【工程落地】** 你没有把高频代码变动混在 Socket.io 里，而是用 `y-websocket` 建立了一条专门的通道：

  ```js
  const ydoc = new Y.Doc(); // 创建本地数学链表
  // 1. 挂载浏览器本地数据库，做离线历史合并树
  const indexeddbProvider = new IndexeddbPersistence(`room-${roomId}`, ydoc);
  // 2. 建立 wss:// 协同专线
  const provider = new WebsocketProvider(import.meta.env.VITE_YJS_URL, roomId, ydoc);
  ```

##### 阶段 B：涂抹胶水，无缝接管 Monaco (Binding)

- **【源码出处】** `frontend/src/App.jsx`

- **【工程落地】** Yjs 是纯数据结构，Monaco 是 UI 编辑器，两者原本互不相识。你使用了 `MonacoBinding` 将它们死死绑定在一起。并且使用了 `Awareness` 协议实现了彩色光标与动态昵称：

  ```js
  // 向 Awareness 协议注入自定义身份（用于渲染别人屏幕上的光标名字）
  provider.awareness.setLocalStateField('user', { name: '尤雨溪', color: '#...' });
  
  // 涂胶水：把 Yjs 链表、Monaco 模型、光标通道 三合一
  bindingRef.current = new MonacoBinding(ytext, targetCache.model, new Set([editor]), provider.awareness);
  ```

#### 4. 面试死磕防线与防御话术

当面试官看到你的简历上写了“基于 CRDT 算法的 Yjs 协同”，他一定会抛出这道极具杀伤力的深水炸弹：

**⚔️ 面试官必杀追问：“你既然用了 WebSocket，那如果用户 A 拿着笔记本进了电梯，断网了 5 分钟，他在电梯里敲了几百行代码。等他出电梯重新连上 WebSocket 时，这 5 分钟的草稿会丢失吗？或者会覆盖别人在这 5 分钟里写的代码吗？请说明底层合并原理。”**

> **🛡️ Senior 架构师防御话术：** “绝对不会丢失，也不会发生全量覆盖冲突。
>
> 1. **离线守护**：首先，我在 `useWorkspaceSocket` 中挂载了 `y-indexeddb`。断网期间，用户所有的敲击都会被 Yjs 转化为包含 `(ClientID, Clock)` 的操作日志，默默追加到浏览器的 IndexedDB 中。
> 2. **状态向量握手**：出电梯重连的一瞬间，Yjs 触发『双向握手协议』。客户端会从 IndexedDB 读取并计算出一个轻量级的 **状态向量 (State Vector)** 发给服务器（告诉服务器：这是我目前的进度）。
> 3. **无损编织**：服务器收到后，计算差分并发回缺失数据；同时客户端也会把电梯里写的『私有增量日志』发给服务器。因为 CRDT 的特性，这些字符节点会根据 ClientID 确定性地排序，像拉链一样在数学层无缝咬合（Woven），实现零冲突的最终一致性。”
