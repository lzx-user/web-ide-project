作为面试全栈或前端岗位的亮点，你的 Node.js 后端已经具备了很好的骨架（路由分离、JWT 鉴权、Socket 房间隔离）。但从“工程级健壮性”的角度来看，**目前的后端代码如果直接上线，有几个很容易导致服务器宕机的隐患。**

外企极其看重系统的**稳定性 (Stability)** 和**边缘边界处理 (Edge Cases)**。我们重点优化以下几个点：

#### ⚠️ 隐患一：Node.js 子进程奔溃未捕获（致命级）

在 `backend/index.js` 的 `executeCode` 事件中，你使用了 `spawn` 来运行代码：

JavaScript

```
const child = spawn('node', [filePath], { timeout: 5000 });
```

**问题所在**：如果用户代码导致系统级的 spawn 错误（例如内存爆了，或者某些极端语法导致 node 进程直接 crash，而不是代码内部报错），`child` 会抛出一个 `error` 事件。**如果没有监听这个事件，Node.js 会直接抛出未捕获的异常 (Uncaught Exception)，导致整个后端服务器崩溃。**

#### ⚠️ 隐患二：并发文件名冲突

在生成临时文件时，你使用了 `Date.now()`：

JavaScript

```
const filePath = path.join(tempDir, `stream_${Date.now()}.js`);
```

**问题所在**：在毫秒级别内，如果有两个用户同时点击“运行”，它们生成的文件名是一模一样的！后写入的会覆盖前一个的代码，导致某个人运行的代码不是自己的。

#### ⚠️ 隐患三：安全沙箱缺失 (面试高频考点)

虽然加了 `timeout: 5000`，但用户依然可以通过写 `fs.rmSync('/', { recursive: true })` 这样的恶意代码来删除你服务器上的文件！

### 🛠️ 后端优化行动指南

这些问题我们不一定都要做极其复杂的改造，但要在代码里体现出你的**防御性编程思维**。请按照以下步骤替换你的后端代码：

#### 第一步：修复并发冲突与进程崩溃风险 (`backend/index.js`)

在文件顶部引入 Node.js 原生的 `crypto` 模块用来生成唯一 ID，然后完善 `executeCode` 的监听。

JavaScript

```react
// backend/index.js (只展示需要修改的 executeCode 部分)
// 记得在文件顶部加上: const { randomUUID } = require('crypto');

  // 监听：流式执行代码
  socket.on('executeCode', async (payload) => {
    const codeStr = typeof payload === 'object' ? payload.code : payload;

    // 修复 1：使用 UUID 替代 Date.now()，彻底杜绝高并发下的文件名冲突！
    const filePath = path.join(tempDir, `stream_${randomUUID()}.js`);

    try {
      io.to(roomId).emit('executionStarted');
      await fs.promises.writeFile(filePath, codeStr, 'utf8');

      const child = spawn('node', [filePath], { timeout: 5000 });

      // 修复 2：极其关键的 error 事件兜底！防止进程级别报错导致整个后端宕机
      child.on('error', (err) => {
        console.error('子进程启动失败:', err);
        io.to(roomId).emit('terminalError', `\n[系统错误: 无法启动运行环境 ${err.message}]`);
      });

      child.stdout.on('data', (data) => {
        io.to(roomId).emit('terminalOutput', data.toString());
      });

      child.stderr.on('data', (data) => {
        io.to(roomId).emit('terminalError', data.toString());
      });

      child.on('close', async (exitCode) => {
        io.to(roomId).emit('executionFinished', exitCode);
        if (fs.existsSync(filePath)) {
          await fs.promises.unlink(filePath);
        }
      });

    } catch (err) {
      io.to(roomId).emit('terminalError', `服务器内部错误: ${err.message}`);
    }
  });
```

#### 第二步：加固伪终端销毁逻辑 (`backend/src/pty/PtyManager.js`)

当用户疯狂刷新页面时，会频繁触发 `disconnect`。如果终端进程刚好自己退出了，这时候再去 `.kill()` 会抛错。我们需要加上 `try...catch`。

JavaScript

```react
// backend/src/pty/PtyManager.js (替换 destroy 方法)
  
  // 销毁方法：用户断开时一定要清理，防止服务器卡死
  destroy() {
    this.socket.removeAllListeners('terminal-in');
    try {
      // 优雅退出进程，防止孤儿进程占用系统资源
      if (this.ptyProcess) {
        this.ptyProcess.kill();
      }
    } catch (error) {
      console.log('PTY 进程已销毁或无需销毁');
    }
  }
```
