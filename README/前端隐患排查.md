### 1. 状态管理 (useIDEStore) 隐患排查

你在 Zustand 中的状态切分得很漂亮，但存在一个容易引发应用崩溃的致命细节：

**🚨 致命隐患：未初始化的嵌套状态** 在 `useIDEStore.js` 的 `addLog` 方法中：

JavaScript

```
addLog: (filename, type, text) =>
  set((state) => {
    // 隐患点：如果 state.terminalLogsMap 是 undefined，这里会报错
    const currentLogs = state.terminalLogsMap[filename] || [];
    return {
      terminalLogsMap: {
        ...state.terminalLogsMap,
        [filename]: [...currentLogs, { id: Date.now(), type, text }],
      },
    };
  }),
```

**修复方案**：必须在 Store 的初始化部分（State 定义区）声明 `terminalLogsMap: {}`，否则第一次调用 `addLog` 时，`state.terminalLogsMap` 为 `undefined`，解构或读取属性会直接触发 TypeError 导致白屏。

### 2. 核心链路 (App.jsx) 内存泄漏与闭包陷阱检查

`App.jsx` 作为整个 IDE 的调度中枢，承载了极重的生命周期管理。这里有几个典型的内存泄漏风险点需要处理：

**⚠️ 风险一：Lodash 防抖函数的内存泄漏** 你使用了 `useCallback` 配合 `debounce` 来处理代码同步：

JavaScript

```
const debouncedEmitCode = useCallback(
  debounce((code) => { ... }, 500),
  []
);
```

**问题所在**：当组件卸载（比如用户点击退出房间）时，如果恰好有一个代码同步事件还在 500ms 的等待期内，它依然会在组件销毁后触发，这不仅会导致网络冗余请求，还可能引发 React 状态更新警告。 **修复方案**：在组件卸载时手动 `cancel()` 掉未执行的防抖函数。

JavaScript

```
useEffect(() => {
  return () => {
    debouncedEmitCode.cancel(); // 清理防抖定时器
  };
}, [debouncedEmitCode]);
```

**⚠️ 风险二：Socket 事件的粗暴卸载** 在监听 Socket 的 `useEffect` 的清理函数中：

JavaScript

```
return () => {
  currentSocket.off('terminalOutput', handleOutput);
  // ...
  currentSocket.off('codeChange'); // 👈 隐患
  currentSocket.off('initCodePackage'); // 👈 隐患
};
```

**问题所在**：调用 `.off('eventName')` 如果不传入具体的回调函数引用，会**清空该事件下的所有监听器**。如果未来项目中其他组件也监听了 `codeChange`，这里的卸载会把别人的监听器也一并误杀。 **修复方案**：务必将监听器提取为具名函数，并在 `off` 时精确传入。

JavaScript

```
const handleCodeChange = (payload) => { ... };
currentSocket.on('codeChange', handleCodeChange);
// 清理时：
currentSocket.off('codeChange', handleCodeChange);
```

**⚠️ 风险三：Monaco Editor 模型 (Model) 内存泄漏** 你在 `fileCacheMap` 中缓存了大量 Monaco Model：

JavaScript

```
const newModel = monacoRef.current.editor.createModel(...)
```

**问题所在**：Monaco Editor 创建的 Model 是驻留在浏览器内存中的。当用户退出房间时，虽然 React 组件销毁了，但这些 Model 实例依然存在，频繁进出房间会导致内存暴涨。 **修复方案**：在退房逻辑或组件整体卸载的 `useEffect` 中，销毁所有缓存的 Model。

JavaScript

```
useEffect(() => {
  return () => {
    if (monacoRef.current) {
      monacoRef.current.editor.getModels().forEach(model => model.dispose());
    }
  };
}, []);
```

### 3. 代码质感与架构优化 (提升至外企标准)

为了让代码更具专业素养，建议在以下几个方面进行重构：

**✨ 优化一：消除“魔法字符串 (Magic Strings)”** `App.jsx` 中散落着大量像 `'ide_token'`、`'draft-${roomId}-${filename}'` 这样的硬编码字符串。一旦拼写错误，极难排查。 **建议**：新建一个 `constants.js` 或直接在文件顶部定义配置字典：

JavaScript

```
const STORAGE_KEYS = {
  TOKEN: 'ide_token',
  ROOM_ID: 'ide_roomId',
  IS_JOINED: 'ide_isJoined',
  getDraftKey: (roomId, filename) => `draft-${roomId}-${filename}`
};
```

在代码中替换为 `localStorage.getItem(STORAGE_KEYS.TOKEN)`，这会让代码极具可维护性。

**✨ 优化二：抽离巨型 useEffect (自定义 Hook 化)** `App.jsx` 目前有超过 300 行，其中 Socket 监听的 `useEffect` 占据了巨大篇幅。业务逻辑与视图层开始耦合。 **建议**：将 Socket 通信逻辑抽离为专属的 Custom Hook，例如 `useWorkspaceSocket.js`：

JavaScript

```
// 在 App.jsx 中只需要这样调用：
useWorkspaceSocket({
  roomId,
  currentSocket,
  fileCacheMap,
  lastReceivedCodeRef,
});
```

这样 `App.jsx` 的核心关注点将完美回归到“调度”与“UI 渲染”上。

**✨ 优化三：防御性本地存储访问** 在读取本地历史代码时：

JavaScript

```
const draftCode = localStorage.getItem(`draft-${roomId}-${targetFile}`) || '';
```

虽然这段代码目前没问题，但严谨的做法是包裹 `try...catch`。因为在某些浏览器（如 Safari 的无痕模式）或用户磁盘空间不足时，`localStorage` 的读写可能会直接抛出异常，导致整个页面崩溃。
